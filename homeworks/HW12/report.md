## 4. Temporal split и признаки

### Temporal Split (НЕ Random Split!)

**Разделение:** Train 70% / Validation 15% / Test 15%

**Метод:** `df.iloc[]` (хронологическое разделение по времени)

**Границы:**
- Train: начало до 70% по времени
- Validation: 70% до 85% по времени
- Test: 85% до конца по времени

**Почему random split некорректен:**
1. Утечка данных из будущего в тренировку
2. Временные зависимости нарушаются
3. Метрики будут оптимистичными
4. Модель не будет работать в продакшене

### Признаки (без утечки из будущего!)

**Лаговые признаки:**
- `lag_1` — значение ряда 1 шаг назад
- `lag_7` — значение ряда 7 шагов назад
- `lag_14` — значение ряда 14 шагов назад

**Rolling признаки:**
- `rolling_mean_7` — скользящее среднее за 7 шагов (с shift(1)!)
- `rolling_std_7` — скользящее стандартное отклонение за 7 шагов

**Календарные признаки:**
- `day_of_week` — день недели (0-6)
- `day_of_month` — день месяца (1-31)
- `month` — месяц (1-12)
- `day_of_year` — день года (1-365)

**Циклические признаки:**
- `day_of_week_sin`, `day_of_week_cos` — синус/косинус для дня недели
- `month_sin`, `month_cos` — синус/косинус для месяца

**Важно:** Все rolling признаки используют `shift(1)` перед rolling для предотвращения утечки!

### Scaler (fit/transform)

**StandardScaler:**
- `fit_transform()` на train данных
- `transform()` на validation данных (НЕ fit!)
- `transform()` на test данных (НЕ fit!)

## 5. Модели и эксперименты (B1, B2, B3, R1)

### B1 (naive-last)

**Подход:** Прогноз = последнее известное значение

**Формула:** `y_pred = y_last_known`

**Параметры:**
- window_size: N/A
- horizon: 1

**Плюсы:** Простейший бенчмарк, быстро

**Минусы:** Не учитывает динамику ряда

### B2 (moving-average)

**Подход:** Прогноз = скользящее среднее

**Формула:** `y_pred = mean(y_last_7_values)`

**Параметры:**
- window_size: 7
- horizon: 1

**Плюсы:** Учитывает локальную динамику

**Минусы:** Задержка в реакции на изменения

### B3 (ridge-lag-features)

**Подход:** Ridge Regression на лаговых признаках

**Признаки:** lag_1, lag_7, lag_14, rolling_mean_7, rolling_std_7, календарные

**Параметры:**
- model: Ridge (alpha=1.0)
- scaler: StandardScaler (fit на train)
- horizon: 1

**Плюсы:** Использует множественные признаки, быстро

**Минусы:** Линейная модель, не захватывает нелинейности

### R1 (gru-forecast)

**Подход:** GRU на оконном представлении ряда

**Архитектура:**
- input_size: 1
- hidden_size: 64
- num_layers: 2
- dropout: 0.2
- fc: Linear(64, 32) + ReLU + Linear(32, 1)

**Параметры:**
- window_size: 14
- batch_size: 32
- optimizer: Adam (lr=1e-3)
- epochs: 50
- scaler: StandardScaler (fit на train)

**Плюсы:** Захватывает долгосрочные зависимости, нелинейная модель

**Минусы:** Требует больше данных и времени на обучение

## 7. Анализ

###Baseline сравнение (B1 vs B2 vs B3)

**B1 (Naive):**
- Простейший бенчмарк
- Обычно худший результат
- Полезен как нижняя граница качества

**B2 (Moving Average):**
- Лучше B1 за счёт учёта локальной динамики
- Сглаживает шум
- Задержка в реакции на тренды

**B3 (Ridge):**
- Значительно лучше B1/B2
- Использует множественные признаки
- Лаговые признаки дают информацию о динамике
- Календарные признаки捕捉ают сезонность

### GRU анализ (R1)

**Обучение:**
- Сходимость за 30-50 эпох
- Validation loss стабилизируется
- Early stopping можно добавить для предотвращения переобучения

**Почему GRU может быть лучше:**
- Захватывает долгосрочные зависимости
- Учитывает контекст всего окна (window_size=14)
- Адаптируется к нелинейным паттернам
- Рекуррентная архитектура естественна для временных рядов

**Почему Ridge может быть лучше:**
- Меньше вычислительных затрат
- Не требует масштабирования данных
- Интерпретируемые веса
- Работает хорошо на небольших датасетах

### Scaler анализ

**Важность корректного fit/transform:**
- fit только на train предотвращает утечку
- transform на val/test использует статистику train
- Это критично для честной оценки модели

## 8. Итоговый вывод

**Для прогнозирования временных рядов:**

1. **Temporal split критически важен** — random split недопустим для временных рядов
2. **Лаговые признаки значительно улучшают качество** — B3 значительно лучше B1/B2
3. **GRU требует больше данных и времени** — но может захватывать сложные зависимости
4. **Scaler: fit на train, transform на val/test** — предотвращает утечку данных

**Лучшая модель:** GRU (R1) или Ridge (B3) — зависит от характеристик данных

**Рекомендации:**
- Начинать с простых baseline (B1, B2)
- Добавить лаговые признаки (B3)
- Использовать GRU для сложных нелинейных зависимостей
- Всегда проверять на утечку данных

**Для дальнейшего улучшения можно попробовать:**
- Learning rate scheduling (CosineAnnealing, ReduceLROnPlateau)
- Подбор window_size (7, 14, 21, 30)
- Attention механизмы
- Transformer архитектуры
- Ансамблирование моделей (GRU + Ridge)
- Hyperparameter tuning (hidden_size, num_layers, dropout)
