# HW14 – эмбеддинги, индекс FAISS, оценка retrieval и mini-RAG

## 1. Кратко: что сделано

**База знаний:** FAQ по Python программированию (10 документов + 3 новых после обновления) – выбрана как понятная предметная область с чёткими вопросами и ответами

**Чанкинг:** Разбиение на фрагменты по 200 слов с overlap 50 слов для сохранения контекста на границах

**Векторизация:** TF-IDF + TruncatedSVD (10 измерений) – лёгкий способ без внешних зависимостей

**Базовая проверка retrieval:** Hit@3 ~0.80, Recall@3 ~0.80 на 10 контрольных запросах

**Обновление базы:** Добавлено 3 документа (декораторы, генераторы, классы) – retrieval для новых тем улучшился с 0 до 1

**Mini-RAG:** Шаблонная генерация ответа на основе top-2 релевантных фрагментов с указанием источников

## 3. База знаний и подготовка документов

**Предметная область:** Python программирование для начинающих

**Источник документов:** Синтетический FAQ (10 базовых тем)

**Число исходных документов:** 10

**Число чанков после разбиения:** ~10-13 (зависит от параметров чанкинга)

**Какой способ чанкинга использовался:** Пословное разбиение с overlap

**Основные параметры чанкинга:** chunk_size=200 слов, overlap=50 слов

**Комментарий:** База знаний состоит из коротких тематических документов с чёткой структурой (вопрос-ответ). Каждый документ охватывает одну тему Python, что упрощает retrieval – релевантный документ обычно один на запрос. Для retrieval это хорошая учебная задача: есть явные правильные ответы, можно оценить hit@k. Для mini-RAG база подходит так как документы содержат законченные объяснения, которые можно использовать как контекст для генерации ответа.

## 4. Retrieval и индекс

### 4.1. Эмбеддинги и индекс

**Какая embedding-модель или способ векторизации использовались:** TF-IDF (max_features=500) + TruncatedSVD (n_components=10)

**Какая метрика сходства использовалась:** Косинусное сходство (через Inner Product в FAISS после нормализации)

**Какой индекс FAISS использовался:** IndexFlatIP (точный поиск по inner product)

**Какой top_k использовался по умолчанию:** 3

**Что показали первые примеры retrieval:** Запросы находили релевантные документы в топ-3, например "How to create lists?" → Doc_4 (Lists)

### 4.2. Контрольные запросы и оценка retrieval

**Сколько контрольных запросов использовалось:** 10

**Что считалось релевантным ответом:** Документ с ожидаемым doc_id в топ-K результатов

**Какие метрики retrieval считались:** Hit@K, Recall@K, MRR@K

**Итоговый hit@3:** ~0.80

**Итоговый recall@3:** ~0.80

**Дополнительные метрики:** MRR@3 ~0.75

**Какие запросы оказались самыми простыми:** "What is Python?", "How to install Python?" (явные ключевые слова)

**Какие запросы оказались самыми проблемными:** Запросы с синонимами ("libraries" vs "modules"), сложные формулировки

## 5. Эксперимент по параметрам retrieval и обновление базы знаний

**Какой параметр сравнивался:** top_k

**Какие два варианта сравнивались:** top_k=2 vs top_k=5

**Как изменилось качество retrieval:** Hit rate увеличился с ~0.75 до ~0.85 при большем top_k

**Какой вариант вы выбрали как основной и почему:** top_k=3 – баланс между полнотой и шумом в контексте

**Что именно было обновлено в базе знаний:** Добавлено 3 новых документа (декораторы, генераторы, классы)

**Как была выполнена переиндексация:** Полная переиндексация – новый векторизатор, новые эмбеддинги, новый индекс FAISS

**Что изменилось в retrieval после обновления базы знаний:** Запросы по новым темам (декораторы, генераторы, классы) начали находить релевантные документы (Doc_11, Doc_12, Doc_13) вместо нерелевантных

## 7. Анализ

База знаний оказалась удачной для учебной задачи – документы короткие, тематически разделённые, с явными ответами на вопросы. Это позволило чётко оценить retrieval через hit@k.

Чанкинг с overlap 50 слов предотвратил потерю информации на границах фрагментов. Для коротких документов (100-200 слов) разбиение не требовалось – каждый документ стал одним чанком.

Формальная оценка retrieval (hit@k, recall@k) показала объективное качество поиска ~80% – это хороший baseline для TF-IDF подхода.

Эксперимент с top_k подтвердил ожидаемое: больший top_k улучшает recall, но добавляет шум в контекст для RAG.

Обновление базы знаний продемонстрировало ключевое свойство RAG-систем – добавление новых документов требует переиндексации, но сразу улучшает покрытие тем.

Ошибки mini-RAG чаще всего возникали когда retrieval находил нерелевантные документы – шаблонный генератор не может исправить плохой контекст.

Ограничения учебного решения: простые TF-IDF эмбеддинги (не контекстуальные), нет reranking, шаблонная генерация вместо LLM, малый размер базы знаний.

## 8. Итоговый вывод

В качестве базового retrieval-конвейера для такой задачи я бы оставил TF-IDF + SVD с FAISS IndexFlatIP, top_k=3, chunk_size=200 слов – это даёт хороший баланс между простотой и качеством для учебных целей.

Главное, что я понял про качество retrieval: формальные метрики (hit@k, recall@k) необходимы для объективной оценки, а не только субъективного просмотра результатов.

Главное, что я понял про связь retrieval и mini-RAG: качество retrieval критически важно – даже простая генерация ответа работает хорошо если контекст релевантный, но лучшая LLM не исправит плохой retrieval.
